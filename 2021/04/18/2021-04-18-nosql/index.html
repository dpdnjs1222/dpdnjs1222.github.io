[Database] NoSQL 개요

Posted on 2021-04-18 In DB

NoSQL이란 무엇인가?

NoSQL database 주로 non-relational database, 즉 relational table 이외의 포맷으로 데이터를 저장하는 DB를 의미한다. 주의할 점은, NoSQL은 relationship data를 저장하지 못하는 것이 아니라 relational database와는 다른 방식으로 저장한다는 것이다. 오히려 relationship data가 여러개의 테이블로 나눠질 필요가 없기 때문에 SQL와 비교하여 좀 더 쉽게 모델링할 수 있다.

2000년대 후반, storage 비용이 급격히 감소하는 대신 어마어마하게 많은 양과 unstructured 데이터 저장의 필요성이 증가했고 NoSQL은 유연하게 이를 가능하게 했다. 또한, Agile 개발방법론이 대세가 됨에 따라 변하는 요구사항에 쉽게 적응할 수 있는 NoSQL이 각광받았다.

SQL이란 무엇인가?

전통적으로 인기있는 database는 relational database이었다. relational database는 고정된 column과 row를 가지는 테이블에 data가 저장되는 것으로 SQL(Structured Query Language)를 통해 접근된다. 1970년대, storage 비용이 매우 비싼 시기엔 data duplication을 줄이기 위해 db를 normalize했다. 또한 그 시기엔 waterfall development 방식이 주로 사용되었기 때문에 개발 전 복잡한 ER 다이어그램(entity-relationship)을 만들었다. 하지만 이러한 방식은 요구사항이 바뀌는 경우 많은 비용을 유발하였다.

NoSQL 데이터베이스의 종류

  • document database : document(JSON과 유사)에 데이터를 저장, 각 document는 field와 value의 쌍으로 구성되고 value는 문자열부터 객체까지 다양한 타입을 가짐. ex) MongoDB
  • key-value database : key, value 쌍으로 구성, 간단한 query로 방대한 데이터 처리, user preference나 caching에 주로 사용. ex) Redis, DynanoDB
  • wide-column stores : 각 row가 같은 column을 가질 필요 없음(dynamic columns), 방대한 데이터 처리 with predictable query patterns. ex) HBase, Cassandra
  • graph databases : node와 edge 사용, 관계(SNS, fraud detection, recommendation) 파악에 유용. ex) Neo4j, JanusGraph

SQL vs NoSQL

항목 SQL NoSQL
모델 테이블(고정된 열과 행) document, key-value, wide-column, graph
배경 데이터 중복 제거 확장성, 애자일
예시 Oracle, MySQL, Microsoft SQL Server, PostgreSQL MongoDB and CouchDB, Redis and DynamoDB, Cassandra and HBase, Neo4j
스키마 고정 융통성있음
Scaling 수직적 수평적
Multi-Record ACID(atomicity, consistency, isolation, durability) Transactions 지원 일부 지원(MongoDB)
Joins 필요 필요하지 않음
Data to Object mapping ORM(object-relational mapping) ORM 필요 X(mongoDB는 직접적으로 mapping)

NoSQL 장점

  • flexible data models : 요구사항이 변함에 따라 database를 바꾸기 쉬움
  • horizontal scaling : 성능이 더 좋은 서버가 기존 서버를 대체하는 scale-up model이 아닌 기존의 서버 옆에 추가 서버를 배치하는 scale-out model
  • fast queries : SQL은 normalize되어있기 때문에 여러 테이블을 join해야하지만 NoSQL은 query에 최적화되어있음(같이 접근되는 데이터는 같이 저장됨)
  • easy for developers : application에서 사용되는 방식과 같은 구조로 저장되기 때문에 구현이 편리함
